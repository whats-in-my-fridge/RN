## BottomSheet 재오픈 시 높이 축소 + 스크롤 막힘 버그

**날짜:** 2026-03-19

### 문제 상황

`ShelfDetailSheet`의 아이템 목록을 고정 높이 + 스크롤 구조로 만들려 했으나 두 가지 버그가 연속으로 발생했다.

1. `BottomSheetScrollView`에 `height: 500` style을 직접 지정하자, 처음 열 때는 정상이나 시트를 닫고 재오픈하면 높이가 매우 좁아지는 버그 발생. 스크롤도 당겨지다 멈추는 현상.
2. `snapPoints={["60%"]}`를 도입해 `enableDynamicSizing`을 대체하려 했으나, 시트가 처음에 매우 낮은 높이로 열렸다가 사용자가 직접 올려야 하는 버그 발생.
3. `enableDynamicSizing: false`를 명시적으로 추가하자 60%에서 즉시 열리게 됐으나, 스크롤이 여전히 조금 당겨지다 멈추는 증상 지속.

### 해결 방법

근본 원인은 `BottomSheetView`(동적 사이징 전용)와 `BottomSheetScrollView`의 중첩 구조였다. `BottomSheetView`는 내부 콘텐츠 크기를 시트에 계속 보고하는 컴포넌트로, `enableDynamicSizing`과 한 쌍으로 설계되어 있다. 이 안에 `BottomSheetScrollView`를 넣으면 스크롤 가능 영역의 높이 계산이 꼬인다.

`snapPoints` 사용 시에는 `BottomSheetView` 대신 일반 `View`로 분기하여 해결했다.

```tsx
// BottomSheet.tsx — snapPoints 여부에 따라 컨테이너 분기
{snapPoints ? (
  <View style={bottomSheetStyles.content}>{children}</View>
) : (
  <BottomSheetView style={bottomSheetStyles.content}>{children}</BottomSheetView>
)}
```

추가로 `BottomSheetModal`에서 `enableDynamicSizing`의 기본값이 `true`이므로, `snapPoints` 사용 시 반드시 `enableDynamicSizing: false`를 명시해야 한다.

```tsx
const sizeProps = snapPoints
  ? { snapPoints, index: 0, enableDynamicSizing: false }
  : { enableDynamicSizing: true, maxDynamicContentSize };
```

### 프론트엔드 개발자로서 더 발전할 수 있는 방안

- `@gorhom/bottom-sheet`의 컴포넌트별 설계 의도를 사전에 파악하는 습관이 필요하다. `BottomSheetView`는 동적 사이징 전용, `BottomSheetScrollView`는 고정 snapPoints 전용이라는 규칙이 있다.
- 라이브러리의 컴포넌트를 조합할 때 "이 컴포넌트는 어떤 맥락에서 사용되도록 설계되었는가"를 공식 문서나 소스에서 먼저 확인하면 중첩 구조 충돌을 예방할 수 있다.
- prop 기본값(예: `enableDynamicSizing: true`)은 명시하지 않아도 동작하므로 놓치기 쉽다. 사용 방식을 바꿀 때는 관련 기본값을 함께 검토하는 것이 좋다.

## 냉장고 보드 칩 영역에서 마지막 칩이 부분적으로 잘려 보이는 버그

**날짜:** 2026-03-19

### 문제 상황

`DualShelfSection`(좌우 분할 선반)에서 아이템이 5개 이상이 되면 `flex-wrap`으로 줄바꿈된 칩이 섹션 할당 높이를 초과하며, `overflow-hidden`이 칩 중간을 잘라 마지막 칩이 희미하게 반쪽만 보이는 현상이 발생했다.

최초 접근으로 `MAX_VISIBLE_CHIPS` 개수 제한을 시도했으나, 칩 너비가 이름 길이에 따라 가변적이라 개수와 실제 줄 수가 일치하지 않아 이름이 길면 2줄을 넘기거나 짧으면 공간을 낭비하는 문제가 있었다.

### 해결 방법

`ItemChip`의 렌더링 높이를 스타일에서 직접 역산하여 칩 컨테이너에 고정 픽셀 높이를 적용했다.

- `ItemChip` 높이: `py-[3px]`(6px) + `leading-[17px]`(17px) = **23px**
- 행 간 gap: `gap-1.5` = **6px**
- 2줄 기준 고정 높이: 23 × 2 + 6 = **52px**

```tsx
// CHIP_HEIGHT * 2 + CHIP_ROW_GAP = 52px — overflow-hidden이 정확히 행 경계에서 클리핑
const CHIP_HEIGHT = 23;
const CHIP_ROW_GAP = 6;
const CHIPS_AREA_HEIGHT = CHIP_HEIGHT * 2 + CHIP_ROW_GAP;

<View
  className="flex-row flex-wrap gap-1.5"
  style={{ height: CHIPS_AREA_HEIGHT, overflow: "hidden" }}
>
```

이 방식은 칩 개수나 이름 길이에 무관하게 항상 정확히 2줄 경계에서 잘리므로 부분 노출이 구조적으로 불가능하다.

### 프론트엔드 개발자로서 더 발전할 수 있는 방안

- `overflow: hidden`은 픽셀 단위로 자르기 때문에 컴포넌트 경계를 인식하지 못한다. 콘텐츠 단위로 자르고 싶다면 컨테이너 높이를 컴포넌트 단위의 배수로 명시적으로 지정해야 한다.
- 디자인 토큰(`py-[3px]`, `leading-[17px]`)에서 역산한 픽셀 값을 상수로 관리하면, 나중에 칩 스타일이 바뀔 때 상수만 수정하면 되어 유지보수가 용이하다.
- `flex-wrap` 레이아웃은 콘텐츠 가변성으로 인해 높이가 예측 불가능하므로, 고정된 시각적 영역이 필요한 경우 이 접근법을 표준 패턴으로 활용할 수 있다.

## Android 시스템 네비게이션 바와 BottomSheet 콘텐츠 겹침

**날짜:** 2026-03-19

### 문제 상황

Android 기기에서 `BottomSheetModal`을 열면 하단 아이템 행이 시스템 네비게이션 바(버튼식 또는 제스처 바)와 겹쳐 가려지는 현상이 발생했다. `BottomSheetModal`이 시스템 UI 영역을 인식하지 못하고 전체 화면 높이 기준으로 렌더링하기 때문이다.

### 해결 방법

`useSafeAreaInsets().bottom`을 `BottomSheetModal`의 `bottomInset` prop으로 전달하여 시트 포지션 자체를 네비게이션 바 높이만큼 올렸다.

```tsx
const { bottom: bottomInset } = useSafeAreaInsets();

<BottomSheetModal
  bottomInset={bottomInset}
  ...
>
```

`SafeAreaView`로 내부 콘텐츠를 감싸는 방식보다 정확한 이유는, `bottomInset`은 시트의 포지션 및 snap point 계산 자체에 반영되기 때문이다.

### 프론트엔드 개발자로서 더 발전할 수 있는 방안

- Android와 iOS 모두 시스템 UI(네비게이션 바, 홈 인디케이터)가 앱 콘텐츠 위에 올라오므로, 바텀시트·모달 같은 오버레이 컴포넌트를 구현할 때는 항상 `SafeAreaInsets`를 고려해야 한다.
- `SafeAreaView`는 내부 padding을 추가하는 방식이고, `bottomInset` prop은 컴포넌트 위치 자체를 조정하는 방식이다. 오버레이 컴포넌트에서는 후자가 더 정확하고 안정적이다.
- 라이브러리가 `bottomInset` 같은 전용 prop을 제공한다면 직접 inset을 계산하지 말고 해당 prop을 우선 사용하는 습관을 들이자.
