# CapstonePR — Capstone PR 생성 스킬

whats_in_my_fridge 프로젝트의 PR 템플릿에 맞춰 PR을 생성한다.
base 브랜치는 항상 `develop`이며, 이슈 번호는 브랜치명에서 자동 추출한다.

---

## Step 1 — 브랜치 정보 수집 (병렬 실행)

다음 명령을 **동시에** 실행해 현재 브랜치 상태를 파악한다:

```bash
git branch --show-current
git log develop..HEAD --oneline
git diff develop...HEAD --stat
git diff develop...HEAD
git status
```

- 브랜치명 패턴 예시: `feat/#56-fridge-page`, `fix/#23-crash`, `refactor/#49-route`
- 브랜치명에서 이슈 번호(`#N`)를 추출한다. 패턴이 없으면 사용자에게 물어본다.
- `git log` 로 커밋 제목들을 확인해 변경의 흐름을 파악한다.
- `git diff --stat` 로 어떤 파일이 얼마나 변경됐는지 파악한다.
- `git diff` 전체 diff로 실제 변경 내용을 파악한다.

---

## Step 2 — PR 템플릿 작성

아래 템플릿의 각 섹션을 채운다. **반드시 한국어**로 작성한다.

```markdown
## 요약

- <이 PR이 해결하는 문제 또는 제공하는 가치를 1~3개 불릿으로. "무엇을 추가했다" 나열이 아니라 "왜", "어떤 목적"으로 중심>

## 관련 이슈

<!-- PR 머지 시 자동으로 이슈가 닫힙니다. 예: Closes #123 -->
- Closes #<브랜치명에서 추출한 이슈 번호>

## 작업 세부사항

- <FSD 레이어(pages / widgets / features / entities / shared / app) 기준으로 변경 내용을 분류해 불릿 나열>
- <새로 추가된 파일, 수정된 파일, 삭제된 파일을 구분해서 명시>
- <컴포넌트·훅·타입·유틸 등 구체적인 단위까지 기술>
- <스텁(stub) 상태인 파일이 있다면 "스텁 상태" 명시>

## 참고사항

- <리뷰어가 알아야 할 설계 결정 이유, 트레이드오프, 알려진 한계점>
- <후속 이슈가 있다면 해당 이슈 번호 언급>
- <없으면 빈 줄로 남길 것>
```

### 각 섹션 작성 지침

**요약**
- "무엇을 했다" 나열이 아닌, 이 PR의 목적·가치·문제 해결 중심
- 예) "냉장고 아이템 목록 조회 API 연동을 위한 FSD 구조 스텁을 생성해 이후 구현의 토대를 마련한다"

**관련 이슈**
- `Closes #N` 형식 고정. 이슈 번호를 모를 경우 사용자에게 묻는다.

**작업 세부사항**
- FSD 레이어 단위로 묶어서 작성한다:
  - `pages/xxx` — 역할 설명
  - `widgets/xxx` — 역할 설명
  - `features/xxx` — 역할 설명
  - `entities/xxx` — 역할 설명
  - `shared/xxx` — 역할 설명
- 파일 단위가 아닌 **의미 단위**(컴포넌트, 훅, 타입 등)로 기술한다.
- FSD 레이어 의존성 방향 준수 여부를 확인해 특이사항이 있으면 명시한다.

**참고사항**
- 설계상 선택지와 그 이유, 리뷰 시 집중해야 할 포인트
- 스텁 파일이 있다면 "실제 구현은 이후 이슈(#N)에서 진행 예정" 형식으로 기술
- 없으면 비워둔다 (섹션 자체는 유지)

---

## Step 3 — 사용자 확인

채운 PR 템플릿 전문을 사용자에게 출력하고, **"이 내용으로 PR을 생성할까요?"** 라고 확인을 받은 뒤 생성한다.
수정 요청이 있으면 반영 후 재확인한다.

---

## Step 4 — PR 생성

확인이 완료되면 아래 형식으로 PR을 생성한다:

```bash
gh pr create \
  --base develop \
  --title "[<prefix>/#<N>] <한국어 핵심 내용 1줄>" \
  --body "$(cat <<'EOF'
<Step 2에서 작성한 템플릿 전문>
EOF
)"
```

### 제목 규칙
- prefix는 브랜치명에서 추출: `feat`, `fix`, `refactor`, `chore`, `docs` 등
- 예시: `[feat/#56] 냉장고 페이지 FSD 구조 스텁 생성`
- 예시: `[fix/#23] 스캔 탭 무한루프 수정`
- 예시: `[refactor/#49] 라우트 구조 리팩토링`

### 주의사항
- `--base develop` 은 절대 생략하지 않는다
- PR 생성 후 반환된 **PR URL을 반드시 출력**한다
- `git push` 가 안 된 상태라면 사용자에게 먼저 push를 요청한다 (`git push -u origin <브랜치명>` 명령어 안내)
- 이모지 사용 금지
